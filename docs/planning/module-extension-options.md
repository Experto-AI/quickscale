# Module Extension Options: Standardization Plan

> **You are here**: [QuickScale](../../START_HERE.md) → [Docs](../index.md) → **Planning** → Module Extension Options
> **Related docs**: [Decisions](../technical/decisions.md) | [Scaffolding](../technical/scaffolding.md) | [User Manual](../technical/user_manual.md) | [Competitive Analysis](../overview/competitive_analysis.md) | [QuickScale Overview](../overview/quickscale.md)

## Goal

Define a standard, Django-native way for QuickScale modules to be extended in generated projects while preserving QuickScale's core product promise:

- QuickScale provides reusable backend modules plus a showcase theme.
- The user customizes the frontend freely.
- The user can extend the backend for project-specific behavior.
- Installed modules should still be able to receive future upstream updates.
- User-owned extension code should remain isolated enough that module updates are adoptable rather than destructive.

This document evaluates the current state, reviews the main architectural alternatives, compares QuickScale against relevant competitors and Django ecosystem references, and makes a concrete standardization decision.

## Executive Summary

QuickScale should **not** standardize on "every module must expose a base class" and should **not** build a universal hook framework as the primary extension model.

The recommended standard is a **layered Django-native extension contract**:

1. **Project-owned extension app by default** for custom signals, admin wiring, orchestration, and glue code.
2. **Stable module-owned extension surfaces** chosen from a small approved set:
   - settings contract
   - template override paths
   - helper/service APIs
   - signals/events
   - admin base classes where useful
   - abstract base models only where the domain genuinely requires subclassing
3. **Explicit fork mode** for deep customization that intentionally diverges from upstream, with the user accepting merge work during updates.

This is the most Django-native and update-compatible model. It also matches the strongest patterns seen in the ecosystem: project-level overrides, explicit registration, package-level contracts, and selective subclassing rather than a mandatory inheritance hierarchy.

## Decision Criteria

The chosen standard should satisfy all of the following:

| Criterion | Why it matters |
| --- | --- |
| Upgrade safety | QuickScale's differentiator is shared updates across projects. |
| Django fit | The architecture should feel like Django, not like a custom framework users must learn first. |
| Cross-module consistency | Modules can differ in behavior, but extension contracts should be recognizable and documented consistently. |
| Applicability across module types | Listings, auth, storage, notifications, and backups do not all have the same extension shape. |
| Low documentation ambiguity | AI assistants and humans should be able to follow one clear rule set. |
| Stable ownership boundaries | It should be obvious which files QuickScale owns and which files the user owns. |
| Future marketplace compatibility | A later package ecosystem should be able to rely on formal contracts, not accidental patch points. |

## Product Context

QuickScale already has a strong ownership model in principle:

- **Themes** are one-time scaffolding. The user owns them immediately and QuickScale does not promise ongoing theme updates. See [decisions.md](../technical/decisions.md).
- **Modules** are long-lived dependencies embedded into the project and updated over time via `quickscale update`. See [user_manual.md](../technical/user_manual.md) and [decisions.md](../technical/decisions.md).
- **Managed integration code** is regenerated by QuickScale into managed files such as `settings/modules.py`, `urls_modules.py`, and `quickscale_managed/*`. See [quickscale_core/src/quickscale_core/module_wiring.py](../../quickscale_core/src/quickscale_core/module_wiring.py).
- **Project-specific backend customization** already has an example shape through `client_extensions`. See [examples/client_extensions/README.md](../../examples/client_extensions/README.md).

That is a good base. The current gap is that the extension story is not yet standardized across modules, and the docs do not clearly separate:

- update-safe customization
- module-specific subclassing
- configuration-only customization
- "you are forking this now" customization

## Current QuickScale State

### System-Level Architecture

QuickScale today uses:

- **Git subtree module distribution** for installed modules and ongoing updates.
- **Declarative module config** in `quickscale.yml` plus applied state tracking.
- **Managed wiring generation** rather than ad hoc patching of user settings and URLs.
- **An opt-in project extension app pattern** rather than a built-in unified extension API.

This means QuickScale already behaves much more like a structured framework plus reusable Django apps than like a plugin platform with dynamic runtime loading. That is consistent with the architectural position documented in [quickscale.md](../overview/quickscale.md): creation-time assembly and static startup registration, not WordPress-style runtime loading.

### Module-by-Module State Snapshot

| Module | Current extension style | Documentation quality | Update-safe today? | Main issue |
| --- | --- | --- | --- | --- |
| `auth` | Settings, template overrides, allauth adapter, signal handler, plus README guidance to edit embedded `User` model directly | Medium | Partly | Direct editing of the embedded user model is not the best update-safe pattern and diverges from mainstream Django guidance for reusable apps. |
| `blog` | Template overrides, feed subclassing, model extension examples | Medium | Mostly | Extension contract exists but is not formalized; some examples blur proxy-model and extra-field customization. |
| `listings` | `AbstractListing`, `AbstractListingAdmin`, template overrides, settings | Good | Yes | Strongest current example, but this pattern is not generalized into a repo-wide rule. |
| `crm` | Config and concrete models/API, extension mostly inferred from Django patterns | Low to medium | Limited | No explicit documented extension contract. |
| `forms` | Data-driven configuration via admin and API, settings, frontend integration | Medium to good | Yes for many use cases | Strong for no-code form definition, weak on documented backend extension hooks. |
| `storage` | Settings plus helper/service functions | Good | Yes | Good contract, but it should be formally classified as a service/helper module rather than a subclassing module. |
| `backups` | Settings, commands, admin surfaces, operational services | Medium for operations, low for extension | Mostly | More an operational module than an extension framework; not currently framed that way. |
| `notifications` | Settings plus service functions and templates | Low to medium | Partly | Useful service surface exists, but the supported extension API is not yet documented as a contract. |
| `social` | Contract-first managed wiring, not yet a full runtime module | Low | Not applicable yet | Too early for a stable extension story. |

### Current Strengths

- Managed wiring already creates a clean boundary between QuickScale-owned integration files and user code.
- Themes are already correctly treated as user-owned frontend code.
- `listings` demonstrates that selective abstract base classes can work well.
- `storage` demonstrates that helper/service APIs can be cleaner than inheritance.
- `client_extensions` demonstrates the right ownership boundary for project glue code.

### Current Problems

1. **No standardized extension taxonomy**. Some modules want subclassing, some want settings, some want service calls, and some just want user-owned glue code.
2. **No standard README section** explaining which extension surfaces are supported and which ones are fork-only.
3. **Auth is the largest architectural mismatch**. The current docs recommend editing embedded module source for user fields, while mainstream Django guidance strongly prefers a project-owned custom user model or at minimum project-owned related models.
4. **No formal fork policy**. Direct edits to `modules/*` are possible, but the docs do not clearly say when that is acceptable and what happens at update time.
5. **AI assistant ambiguity**. An AI assistant can infer the right approach in many cases, but the repo does not yet give it one authoritative extension contract to follow.

## What Is Standard In Django?

### Short Answer

The standard Django way to extend reusable apps is **not** "every app exposes a base class and the user inherits from it."

The common, mainstream Django patterns are:

- project-level settings configuration
- template overriding
- app registration through `INSTALLED_APPS`
- `AppConfig.ready()` plus signals for startup wiring
- abstract base models when a model family genuinely benefits from subclassing
- admin subclassing when admin behavior should be reused
- helper/service APIs for reusable logic
- swappable model references for project-owned foundations such as `AUTH_USER_MODEL`
- project-owned apps for custom business logic

What is **not** considered the mainstream primary pattern is expecting users to edit the source of an embedded reusable app directly as their normal customization flow.

### Django-Native Extension Patterns

| Pattern | Standard in Django? | Typical use |
| --- | --- | --- |
| Settings contract | Yes | Feature flags, defaults, providers, runtime behavior |
| Template overrides | Yes | UI, HTML structure, email templates |
| `AppConfig.ready()` and signals | Yes | Registration and project-specific orchestration |
| Abstract base models | Yes, selectively | Domain models that projects are expected to subclass |
| Admin base classes | Yes, selectively | Reusable admin behavior |
| Service/helper APIs | Yes | Shared logic without inheritance |
| Swappable model references | Yes, selectively | Project-owned foundational models like the user model |
| Global hook registry | Sometimes, but framework-specific | Used by specialized frameworks like Wagtail, not the default pattern for all Django apps |
| Direct edits to third-party app code | No as a primary strategy | Acceptable only as an intentional fork/divergence |

### Important Django Guidance for QuickScale

The most important planning implication is the standard Django position on reusable apps and custom users:

- reusable apps should generally **reference** `settings.AUTH_USER_MODEL`
- reusable apps should generally **not own** the project's custom user model

That does not mean QuickScale must never ship auth-related models. It does mean QuickScale should treat the project's primary user model as a project-level foundation, not as a normal updateable module customization surface.

### Planning Implication

QuickScale should align with Django by treating extension as a **menu of patterns**, not one universal inheritance rule.

## Competitor And Ecosystem Benchmark Snapshot

The internal competitor set is already documented in [competitive_analysis.md](../overview/competitive_analysis.md) and [packages.md](../overview/packages.md). An external research pass across Django and adjacent ecosystem references reinforces the same conclusion: **there is no dominant "base class for every module" pattern across the market.**

### Competitor / Reference Comparison

| Product | Distribution model | Primary customization model | Upgrade model | QuickScale takeaway |
| --- | --- | --- | --- | --- |
| SaaS Pegasus | Generated project | Settings, template overrides, adapters, direct code edits | Managed upstream pull requests / manual merges | Strong generated-project UX, but still merge-heavy when user code and upstream code overlap. |
| Cookiecutter Django | One-time generation | Settings, direct project edits, standard Django overrides | No built-in upgrade mechanism | Good baseline for project ownership, but not enough for QuickScale's shared-update promise. |
| Wagtail | Reusable packages | Hook registry, template overrides, settings, model subclassing | Normal package upgrades | Hooks work well for framework-level extensibility, especially admin/editor concerns. |
| Django CMS | Reusable packages | Plugins, apphooks, template overrides, settings | Normal package upgrades | Explicit registration is powerful for CMS-style composition, but not every module needs plugin machinery. |
| django-Oscar | Reusable packages with app forking | Dynamic class loading, selective app forking, settings, templates | Package upgrades for untouched apps, manual merges for forked apps | Excellent model for an explicit "forked customization" escape hatch. |

### What Competitors Actually Validate

The market validates these patterns:

- **Generated code is easy to own, but hard to keep aligned with upstream**.
- **Reusable packages with explicit contracts are easier to update**.
- **Framework-level hooks are useful when there is a real shared registry problem**.
- **Template overrides and settings remain universal**.
- **Selective forking is a valid escape hatch**, but it should be explicit.

The market does **not** validate these claims:

- every module should expose a base class
- every module should implement a hook registry
- user customization should primarily happen by editing vendor app code directly

## Available Architectural Alternatives

### Option A: Keep The Current Mixed Approach

Description:

- let each module evolve its own extension style
- keep `client_extensions` as an optional pattern
- document the current state without stronger standardization

Pros:

- minimal immediate work
- preserves module-specific freedom

Cons:

- keeps AI and human users guessing
- documentation remains uneven
- update-safe customization remains unclear
- auth remains architecturally inconsistent

Assessment:

This is not enough for QuickScale's long-term module-update promise.

### Option B: Universal Hook Framework Across All Modules

Description:

- every module exposes named hooks
- projects register handlers in a common hook registry
- customization is driven by events and callbacks

Pros:

- consistent mental model
- good for framework-level extensibility
- good for third-party marketplace scenarios later

Cons:

- over-engineered for many modules
- not the default Django pattern
- high maintenance burden and versioning surface
- weak fit for model schema extension and template ownership

Assessment:

Useful in specific areas, but too broad as the primary extension model.

### Option C: Base Class For Every Module

Description:

- each module must expose one or more base classes
- the user extends those classes in project code

Pros:

- easy to explain in theory
- works well for a subset of domain-model modules

Cons:

- poor fit for operational or service-oriented modules like storage, notifications, or backups
- pushes inheritance where composition would be simpler
- not the standard Django answer for reusable apps in general
- risks fragile schema coupling and migration complexity

Assessment:

Good for modules like listings. Wrong as a universal rule.

### Option D: Swappable-Everything / Deep Pluggability

Description:

- make models, services, and views widely swappable through settings or registries
- aim for maximum pluggability from day one

Pros:

- very flexible
- theoretically strong for advanced integrators

Cons:

- complex to design and test
- easy to over-abstract
- creates a large compatibility surface for every release
- unnecessary for many modules

Assessment:

Too expensive as the default architecture. Use only where Django itself already establishes the pattern, such as project-owned foundational models.

### Option E: Layered Django-Native Contract

Description:

- generate a project-owned backend extension app by default
- require every module to document supported extension surfaces from a small approved set
- allow abstract base classes only where the domain calls for them
- use helper/service APIs for operational and integration modules
- standardize template overrides and settings contracts
- define direct edits as explicit fork mode

Pros:

- most Django-native
- compatible with QuickScale's update model
- works across module categories
- easiest to document consistently
- keeps ownership boundaries clear

Cons:

- requires documentation and retrofit work across modules
- still needs module-by-module design judgment

Assessment:

This is the strongest overall fit.

### Option F: Oscar-Style Selective Forking As The Main Strategy

Description:

- keep modules updateable until the user chooses to fork one
- then treat that module as project-owned and manually maintained

Pros:

- honest boundary between updateable and divergent code
- proven by a mature Django ecosystem reference

Cons:

- too heavy as the normal path
- does not solve the need for stable day-to-day customization contracts

Assessment:

Excellent as an escape hatch, not as the primary extension model.

## Recommendation

### Decision

QuickScale should standardize on **Option E: Layered Django-Native Contract**, with **Option F: Explicit Fork Mode** as the deep-customization escape hatch.

### What That Means In Practice

#### 1. Generate A Project-Owned Backend Extension App By Default

QuickScale should stop treating `client_extensions` as a hidden example and instead make a minimal project-owned extension app part of the default generated project.

That app becomes the canonical place for:

- signal registration
- admin customization
- orchestration across modules
- project-only service wrappers
- optional integrations
- glue code that should survive module updates untouched

#### 2. Standardize A Small Approved Extension Surface Set

Every module should declare its supported extension mechanisms using a consistent taxonomy:

| Extension surface | When to use it |
| --- | --- |
| Settings contract | Runtime behavior and provider/config toggles |
| Template overrides | HTML, email, and presentation customization |
| Signals/events | Cross-module reactions and lifecycle hooks |
| Helper/service APIs | Reusable backend behavior without inheritance |
| Admin base classes | Shared admin behaviors for project-owned models |
| Abstract base models | Domain entities intended for project-level subclassing |
| Managed integration files | QuickScale-owned generated wiring only |
| Fork mode | Deep divergence from upstream |

Every module does **not** need to implement every surface.

#### 3. Separate Module Categories Explicitly

| Module category | Preferred extension model | Example modules |
| --- | --- | --- |
| Foundation/auth | Project-owned foundational model plus module integration surfaces | `auth` |
| Domain/content | Optional abstract models and admin bases, templates, settings | `listings`, parts of `blog`, future vertical modules |
| Data-driven | Configuration, admin, API, settings, selected signals | `forms`, parts of `crm` |
| Integration/service | Settings plus helper/service APIs | `storage`, `notifications` |
| Operational | Settings, commands, services, admin actions | `backups` |
| Theme/frontend | User-owned generated code | showcase themes |

#### 4. Define Three Support Tiers For Customization

| Tier | Meaning | Upgrade expectation |
| --- | --- | --- |
| Tier 1: Stable | Project-owned app, settings, template overrides, documented helper/service APIs, documented signals | Should survive normal module updates with minimal or no merge work |
| Tier 2: Structured | Module-specific subclassing such as abstract models or admin bases | Usually survivable across minor updates if the contract is documented and versioned |
| Tier 3: Forked | Direct edits under `modules/<name>/` or equivalent divergence from upstream | User must manually reconcile upstream changes during updates |

This is the clearest way to preserve QuickScale's update promise while remaining honest about deep customization.

## Special Case: Auth Should Change Direction

Auth is the strongest case where QuickScale should move closer to mainstream Django guidance.

### Current Problem

Today the auth module wiring sets `AUTH_USER_MODEL` to a module-owned user class and the README suggests editing the embedded module's `User` model directly.

That creates three problems:

1. it makes a foundational project model live inside updateable module code
2. it encourages the user to modify code that QuickScale later wants to update
3. it diverges from standard Django guidance for reusable apps

### Recommended Direction

QuickScale should move toward this model:

- the project owns the primary custom user model
- auth module code references `settings.AUTH_USER_MODEL`
- auth module continues to own its forms, adapters, templates, URLs, and signal helpers
- project-specific user fields live in project-owned code, not in `modules/auth`

If that transition cannot happen immediately, the documentation should still change now:

- direct editing of `modules/auth/models.py` should be labeled **fork mode**, not the normal extension path
- the preferred short-term pattern should be project-owned related models and project-level signal wiring

## Standard Documentation Contract For Every Module

Every module README should include the same required section:

### Supported Extension Surfaces

Required subsections:

1. **What QuickScale owns**
2. **What the project owns**
3. **Update-safe customizations**
4. **Structured extension points**
5. **Fork-only customizations**
6. **Upgrade expectations**

### Suggested README Template

| Section | Purpose |
| --- | --- |
| Supported extension surfaces | Names the approved customization mechanisms for the module |
| Update-safe examples | Shows the normal path for project developers |
| Fork mode warning | Explains the cost of editing embedded module code |
| Managed files | Lists files regenerated by QuickScale |
| Compatibility notes | Identifies stable APIs versus internal implementation details |

## Proposed Standard By Existing Module

| Module | Proposed standard |
| --- | --- |
| `auth` | Move toward project-owned user model; keep templates, settings, adapters, and signals as module surfaces; classify direct model edits as fork mode |
| `blog` | Keep template overrides and feed subclassing; add clearer documented service/template contract; avoid ambiguous model-extension examples |
| `listings` | Keep `AbstractListing` and `AbstractListingAdmin` as the model example for structured subclassing |
| `crm` | Add documented service/admin/template/settings extension surfaces; avoid implying subclassing unless a real abstract contract is introduced |
| `forms` | Keep admin/data-driven configuration as primary model; document signals/service hooks for custom submission workflows |
| `storage` | Keep helper/service API and settings contract; do not force inheritance |
| `backups` | Keep operational settings and commands; document service layer and explicit non-goals for subclassing |
| `notifications` | Promote `send_notification`-style service contract and template override paths; document stable versus internal APIs |
| `social` | Define the extension contract before the full runtime implementation ships |

## Rollout Plan

### Phase 1: Documentation Standardization

1. Add this architectural decision to `decisions.md`.
2. Create a shared README section template for modules.
3. Update each module README with extension tiers and ownership boundaries.
4. Document `client_extensions` as a first-class project app, not just an example.

### Phase 2: Product Contract Cleanup

1. Generate the project-owned extension app by default.
2. Mark managed files clearly in generated projects.
3. Add CLI or docs guidance around local modifications under `modules/*`.
4. Classify current module APIs as stable, structured, or internal.

### Phase 3: Architectural Corrections

1. Rework auth toward a project-owned user model strategy.
2. Retrofit CRM, Forms, and Notifications with clearer supported surfaces.
3. Add explicit fork-mode guidance inspired by the Oscar model.
4. Add release-note discipline around extension-surface compatibility.

## Compatibility And Update Policy

To make the standard credible, QuickScale should adopt these rules:

1. **Tier 1 surfaces are stable across minor releases** unless explicitly deprecated.
2. **Tier 2 surfaces may evolve, but changes must be called out in release notes.**
3. **Tier 3 fork-mode areas are outside compatibility guarantees.**
4. **Managed files are never user-editable and may be regenerated at any time.**
5. **Themes remain user-owned frontend code and do not participate in the module update guarantee.**

## Final Recommendation

The standard QuickScale answer to "how do I extend a module?" should become:

> Extend QuickScale primarily through a **project-owned extension app** and the module's **documented extension surfaces**. Use abstract base classes only for modules that are explicitly designed for subclassing. Avoid editing embedded module source unless you are intentionally forking that module and accepting manual merge work during updates.

That answer is:

- more Django-native
- more honest about update tradeoffs
- more consistent across module categories
- easier to explain to users and AI assistants
- better aligned with QuickScale's long-term shared-update value proposition
